### 2026-07-02 (14) — Phase 5 BLOCKED: Cognee's deployed config does not actually isolate data per dataset
- Built out full patient-scoping for task #7 (patient CRUD, switcher UI, all API routes moved off `DEMO_PATIENT` onto `requirePatientContext()`/per-patient Cognee datasets, `roster.ts` keyed per-patient, 3-patient seed data) — code changes described below are real and were about to be marked done.
- **Before declaring it done, ran a live cross-patient isolation test** (per this project's established pattern of verifying against the real deployed instance, not assuming): created two patients with two disjoint synthetic diagnoses in two separate Cognee datasets, asked `recall()` against Patient A's dataset about Patient B's diagnosis. **It answered "yes"** — and a follow-up open-ended query ("list every diagnosis") returned facts from Patient B's dataset *and* the original seeded demo patient's (Rina Kapoor) diagnoses, all mixed into one answer, when queried against Patient A's dataset alone.
- **First hypothesis, confirmed wrong via investigation, not assumed:** thought this was an app bug — pulled the live instance's `/openapi.json` and found `RecallPayloadDTO` actually takes a `datasets` array field, not the `datasetName` string the app was sending (which `remember()`/`improve()`/`forget()` correctly use — only `recall()`'s payload shape is different). Fixed `cogneeRecall()` in `src/lib/cognee.ts` to send `datasets: [datasetName]`.
- **Re-ran the isolation test after the fix — still leaked.** Tried again with explicit `datasetIds` (UUIDs, which the schema says "take precedence... resolve by name" — ruling out name-resolution as the cause) pointing at exactly one dataset. **Still returned facts from other patients' datasets and the original demo patient.**
- **Root cause, confirmed:** the deployed Cognee instance has `ENABLE_BACKEND_ACCESS_CONTROL=false` (`deploy/gcp/docker-compose.yml`), set deliberately during Phase 0 for the single-patient hackathon build ("no per-user/dataset isolation needed"). Under that setting, `GRAPH_COMPLETION` retrieval traverses the *entire* shared graph regardless of which dataset(s) are passed to `recall()` — the per-dataset scoping only applies at ingestion (which is why `remember()`/`forget()` *are* correctly dataset-scoped and every previous phase's verification looked correct with only one patient in existence). This was invisible through Phases 0-4 because there was only ever one patient/dataset on the instance; it only surfaces now that a second one exists.
- **Also re-confirmed from Phase 0's own log** (not re-litigated, just re-surfaced because it's now blocking): Cognee's backend access control only supports KuzuDB or Neo4j AuraDB Dev as graph backends — **not** a plain self-hosted Neo4j container, which is what's actually deployed. So turning `ENABLE_BACKEND_ACCESS_CONTROL=true` isn't a config flip; it requires a graph-backend migration.
- Cleaned up all stray test datasets created during investigation (`cogneeForget({dataset: ..., memoryOnly: false})` for each) — confirmed via `cogneeListDatasets()` that only the original `patient_demo_patient_1` dataset remains, untouched.
- **This is a hard blocker for real multi-patient production use, not a code-quality issue** — it means patient data is not actually isolated at the Cognee retrieval layer today, which directly contradicts the multi-tenant isolation goal of Phase 5 and the "Compliance posture" section just added to `CLAUDE.md`. Stopped here rather than marking task #7 done, since shipping a patient switcher UI on top of a shared, unisolated graph would be actively misleading about what the product guarantees. Surfaced to the user for a decision on how to fix the underlying isolation (see conversation) before continuing.
- All the app-side patient-scoping code (queries.ts, patients API routes, roster.ts per-patient keying, all Cognee-calling routes, patient switcher UI) is written and builds/lints clean — it's correct and necessary regardless of how the isolation gap gets resolved, since it's what makes an isolation fix actually take effect once the underlying Cognee config is fixed.

### 2026-07-02 (10) — Phase 4 complete: Forget (correction & archival)
- Checked current implementation first (not assumed): confirmed `src/lib/cognee.ts` had no `forget()` wrapper, no status/roster tracking, and no Patient Summary view existed yet.
- **Investigated the real Cognee `forget()` API before designing** (pulled `/openapi.json` from the live deployed instance and read `ForgetPayloadDTO`): it is document/dataset-scoped only — `dataId` (single data item), `dataset`/`datasetId` (whole dataset), or `everything`. There is no entity-level "forget just this one diagnosis" operation. This directly conflicts with a literal reading of the PRD's forget() examples ("mark a diagnosis ruled out", "mark a medication discontinued"), since the source document for a diagnosis often also contains other still-valid facts that a document-level delete would destroy too.
- **Flagged this to the user before building** (per `CLAUDE.md`'s "design through Cognee, not around it" + "flag deviations before building") and got confirmation to proceed with the recommended approach: track active/ruled-out and current/discontinued status at the app level (doesn't touch the graph destructively), paired with a `remember()`+`improve()` correction narrative so `recall()` itself reflects the change — same pattern as Phase 3's improve() corrections. Reserve literal `forget()` calls for what the API actually supports cleanly: duplicate-document merge and full-document replacement.
- `src/lib/cognee.ts`: added `cogneeForget({ dataId, dataset, memoryOnly })`.
- `src/lib/roster.ts` (new): per-patient roster of diagnoses/medications and their status (`active`/`ruled_out`, `current`/`discontinued`), plus a small document registry (`dataId` + type + date) for duplicate detection. Persisted as a JSON blob via the same `@vercel/blob` store already wired for original-document storage (`status/${patient.id}.json`, fixed pathname + `allowOverwrite: true`, read back via `head()` + `fetch()`) — no new infra dependency. `mergeEntitiesIntoRoster()` upserts by name (case-insensitive) and auto-detects already-resolved/discontinued items from Gemini's own extraction (e.g. seed data's Losartan document, whose dosage field is literally "discontinued").
- **`POST /api/documents/status`** (new): marks a diagnosis ruled-out or medication discontinued — flips the roster status, then `remember()`s a correction narrative ("X's Y diagnosis has been ruled out as of Z...") and `improve()`s the dataset.
- **`POST /api/documents/upload`**: now checks the roster for a same-type+same-date document before remembering; if found, calls `cogneeForget({ dataId: <old>, dataset })` on the superseded document first (the literal, API-supported forget() use case — duplicate merge), then proceeds with `remember()` as before.
